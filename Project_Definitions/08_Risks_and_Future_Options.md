# 风险与未来选项

## 1. 已知风险

### 1.1 上游 GVHMR 偏单机脚本形态

风险：

- 原始代码大量使用 `.cuda()` 和脚本式流程

影响：

- 不适合第一阶段就做任务内多 GPU 并行

策略：

- 第一阶段采用“单 worker 单 GPU”
- 通过任务级并发实现吞吐提升

### 1.2 模型依赖较重

风险：

- worker 镜像较大
- 模型文件难直接放进镜像

策略：

- 模型作为本地只读卷挂载
- 镜像仅包含运行时逻辑

### 1.3 对象存储与本地 scratch 的边界复杂

风险：

- 缓存和最终交付边界容易混淆

策略：

- 最终真源统一进 MinIO
- 本地 scratch 只做临时和局部缓存

### 1.4 上游 GVHMR 缺少 release tag

风险：

- 上游当前没有正式 tag
- 如果直接跟随最新 `main`，上游大改动可能导致 worker 构建或运行失效

影响：

- 新仓库必须自行承担版本钉住策略
- 升级上游时需要显式验证兼容性

策略：

- 使用 git submodule 固定 `GVHMR` commit
- 当前固定 commit：`088caff492aa38c2d82cea363b78a3c65a83118f`
- 升级时先在独立分支验证真实 job，再更新 submodule 指针

## 2. 第一阶段暂不纳入

- 多租户
- 项目空间
- 权限控制
- 配额
- 指标平台
- K8s
- 自动扩缩容

## 3. 未来扩展选项

### 3.1 K8s

前提：

- 控制平面与执行平面边界稳定
- worker 与 API 不依赖宿主机共享文件

### 3.2 更强调度器

可扩展：

- 节点打标
- GPU 类型偏好
- 夜间批处理
- 成本优化策略

### 3.3 更完整 Web 平台

可扩展：

- 权限
- 审计
- 批次模板
- 过滤与搜索

## 4. 架构底线

未来无论如何扩展，以下底线不能被破坏：

- Postgres 仍是元数据真源
- MinIO 仍是输入输出真源
- Redis 不承担长期状态真源角色
- Scheduler 和 Worker 仍需保持独立进程边界
